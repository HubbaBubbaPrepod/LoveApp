1. Контекст и цели
У нас есть готовое Android-приложение LoveApp (Kotlin, Jetpack Compose) с локальной базой данных Room. Серверная часть написана на Node.js (Express) с PostgreSQL, JWT-аутентификацией и FCM для push-уведомлений. Приложение предназначено для пары (два пользователя) и включает следующие разделы: заметки, желания, трекер настроения, активности, календарь месячных, совместные календари, раздел отношений, рисование. В настоящее время рисование уже использует WebSocket (Socket.IO), но остальные данные синхронизируются через REST с периодическими опросами. Мы хотим перевести все данные на real-time синхронизацию, чтобы изменения мгновенно отражались у партнёра, обеспечить офлайн-доступ через Room, повысить производительность, надёжность и масштабируемость, а также добавить современные практики разработки и эксплуатации.

Цели:

Полная real-time синхронизация всех сущностей через WebSocket (Socket.IO).

Офлайн-режим с локальным кэшем (Room) и синхронизацией при восстановлении соединения.

Масштабируемость сервера (горизонтальное масштабирование, балансировка, Redis pub/sub).

Безопасность и защита (rate limiting, helmet, валидация, refresh tokens).

Мониторинг, логирование и алерты (Sentry, Prometheus/Grafana, Winston).

Автоматизация сборки и деплоя (CI/CD, Docker, GitHub Actions, Fastlane).

Тестирование (юнит-тесты, интеграционные тесты, UI-тесты).

Улучшение пользовательского опыта (виджеты, биометрия, экспорт данных, интеграция с системным календарём).

Кэширование, очереди и оптимизация производительности (Redis для кэша, Bull для очередей, Paging 3, Coil, сжатие изображений).

2. Текущее состояние (важно для контекста)
Сервер: Node.js + Express, REST API, PostgreSQL (основные таблицы), FCM для пушей, JWT.

Клиент: Kotlin, Jetpack Compose, Room (локальная БД), Retrofit, Hilt, WorkManager, Glance (виджеты).

Уже реализовано: аутентификация, профили, CRUD для всех разделов, базовые виджеты настроения и активности, рисование через Socket.IO (но только для одного холста, не для всех данных).

3. Детальные требования к новой архитектуре
3.1. Общая схема синхронизации
При запуске приложение загружает все необходимые данные через REST (первоначальная загрузка) и сохраняет в Room.

После успешной аутентификации открывается WebSocket-соединение (Socket.IO). Сервер аутентифицирует соединение по JWT (передаваемому в query или при handshake) и добавляет клиента в комнату, соответствующую паре (например, couple:{coupleKey}). Ключ пары получается из relationship_info.

Все изменения данных (создание, обновление, удаление) сначала применяются к Room (с пометкой syncPending или через отдельную очередь ожидающих действий). Затем через WebSocket отправляется событие data-change, содержащее:

entityType (note, wish, mood, activity, cycle, calendar, event, relationship, artCanvas)

action (create, update, delete)

data (объект сущности или его id для delete)

clientTimestamp (опционально, для разрешения конфликтов).

Сервер валидирует изменение (проверяет права, целостность), сохраняет его в PostgreSQL с обновлённым полем server_updated_at и рассылает это же событие всем остальным клиентам в комнате пары (кроме отправителя).

Получив событие от сервера, клиент обновляет соответствующую таблицу в Room и при необходимости обновляет UI (например, через Flow из DAO).

3.2. Обработка офлайн-режима
Когда соединение отсутствует, все изменения сохраняются в Room с флагом syncPending или добавляются в специальную таблицу outbox.

При восстановлении соединения клиент отправляет на сервер запрос sync-request с указанием последней полученной временной метки (или списка ожидающих действий). Сервер возвращает все изменения, произошедшие после этой метки, в виде массива событий.

Также клиент отправляет накопленные ожидающие действия (если они не были учтены в запросе по времени). Сервер обрабатывает их в порядке очереди, разрешая конфликты по принципу «последняя запись побеждает» (на основе server_updated_at).

При конфликте (например, один и тот же объект изменён офлайн двумя пользователями) побеждает более позднее изменение по серверному времени. Если временные метки совпадают, можно использовать приоритет (например, пользователь с ролью female для цикла).

3.3. WebSocket-события (протокол)
Основные события от клиента к серверу:

authenticate – отправка JWT (если не передаётся при handshake).

data-change – отправка изменения.

sync-request – запрос пропущенных изменений (с параметром sinceTimestamp).

join-room – не требуется, сервер автоматически определяет комнату.

События от сервера к клиенту:

data-change – уведомление об изменении (содержит те же поля, что и от клиента, но serverTimestamp и serverUpdatedAt переопределены).

sync-response – ответ на sync-request, содержит массив событий.

error – сообщение об ошибке (невалидное действие, нет прав и т.д.).

3.4. Роль Redis
Хранение активных соединений: socket:userId -> socketId (или список socketId для пользователя).

Pub/sub для масштабирования: при получении события на одном инстансе сервера, оно публикуется в Redis-канал (например, couple:{coupleKey}). Другие инстансы подписываются на этот канал и отправляют событие своим клиентам в соответствующей комнате.

Кэш: кэшировать часто запрашиваемые данные (профили пользователей, настройки, последние записи) с TTL для снижения нагрузки на PostgreSQL.

Очереди задач (Bull): для асинхронной обработки (генерация превью рисунков, отправка push-уведомлений, массовая рассылка событий).

3.5. Комнаты и подписки
Каждая пара имеет уникальную комнату couple:{coupleKey}. Все события, касающиеся их данных, отправляются в эту комнату.

Для избранных сущностей (например, публичные заметки) можно также использовать индивидуальные комнаты пользователя, но для простоты – общая комната пары.

3.6. Обработка прав доступа
Сервер должен проверять, что пользователь, инициирующий изменение, имеет право на это действие (например, может редактировать только свои заметки, но может видеть заметки партнёра, если они не приватные). Для цикла: только девушка может редактировать, партнёр – только читать.

Эти проверки выполняются при обработке data-change и при REST-запросах.

3.7. REST API (дополнительно)
REST остаётся для:

Первоначальной загрузки данных (GET /api/...).

Загрузки файлов (аватары, изображения к желаниям, холсты).

Авторизации (signup, login, google auth).

Админ-панели.

Операций, не требующих real-time (например, генерация кода партнёра).

REST-эндпоинты должны возвращать данные в формате, совместимом с ApiResponse.

3.8. Кэширование на клиенте (Room)
Все запросы на чтение в UI должны идти через Flow из Room, что обеспечивает автоматическое обновление при изменении локальных данных.

При открытии экрана можно подписываться на обновления через WebSocket (но они уже приходят и обновляют Room).

Для первоначальной загрузки экрана можно показать кэшированные данные, а затем дождаться свежих через REST или WebSocket.

3.9. Разрешение конфликтов
Стратегия: last write wins на основе серверной временной метки (server_updated_at). При сохранении изменения сервер устанавливает server_updated_at = NOW() и сохраняет его. Если приходит более старое изменение (по clientTimestamp), сервер может отклонить его или перезаписать (лучше перезаписать, так как мы считаем серверное время истинным).

Для удалений использовать мягкое удаление: поле deleted_at (timestamp). При получении события delete, сервер проставляет deleted_at, а не удаляет физически. Клиенты получают событие и могут скрыть запись. Для окончательного удаления можно использовать фоновую задачу.

3.10. Производительность и оптимизация
На сервере:

Индексы в PostgreSQL на всех внешних ключах и часто используемых полях (user_id, created_at, updated_at, date и т.д.).

Партиционирование больших таблиц (activity_logs, mood_entries) по дате.

Кэширование в Redis (например, профили пользователей, настройки).

Использование Bull для асинхронных задач (генерация превью рисунков, пуш-уведомления).

Сжатие изображений при загрузке с помощью Sharp.

Балансировка через Nginx, горизонтальное масштабирование Node.js с Redis pub/sub.

На клиенте:

Paging 3 для ленивой загрузки списков (история настроений, активности, заметки).

Coil с настройкой кэша и предзагрузкой.

Accompanist для улучшения UI (pager, insets).

Использование debounce и distinctUntilChanged в потоках.

Оптимизация Compose: remember, derivedStateOf, стабильные типы, key в списках.

3.11. Безопасность
Helmet для Express – установка безопасных HTTP-заголовков.

express-rate-limit – ограничение количества запросов с одного IP.

Валидация входных данных (Joi или express-validator) для всех эндпоинтов.

Refresh tokens – добавить эндпоинт для обновления JWT (с краткосрочным access token и долгосрочным refresh token, хранящимся в БД).

Защита WebSocket – проверка JWT при подключении, отклонение неавторизованных.

Хеширование паролей уже есть (bcrypt).

3.12. Мониторинг, логирование и алерты
Winston + Morgan – логирование запросов и ошибок с ротацией.

Sentry – интеграция с сервером и клиентом для отслеживания ошибок.

Prometheus + Grafana – сбор метрик (количество подключённых сокетов, RPS, время ответа БД, загрузка CPU) и визуализация.

PM2 – для запуска Node.js в production (авто-перезапуск, мониторинг).

3.13. CI/CD и автоматизация
Docker – контейнеризация сервера (Node.js + PostgreSQL + Redis) для локальной разработки и деплоя.

Docker Compose – для локального поднятия всего стека.

GitHub Actions – для автоматического запуска тестов, линтинга, сборки Android APK при пушах.

Fastlane – для автоматической выкладки обновлений в RuStore (или другие магазины).

3.14. Тестирование
Jest + Supertest – для юнит-тестирования API и интеграционных тестов.

MockWebServer – для тестирования сетевого слоя на Android.

Compose UI Testing (createComposeRule()) – для проверки экранов.

Espresso – для сложных UI-тестов.

3.15. Дополнительные пользовательские функции
Виджеты – расширить интерактивность (отметка настроения прямо с виджета).

Биометрическая аутентификация (BiometricPrompt) для быстрого входа.

Экспорт/импорт данных в JSON для резервного копирования.

Интеграция с системным календарём (CalendarContract) для экспорта событий.

Геолокация в реальном времени (если добавить позже) – можно использовать тот же WebSocket.

Аналитика – Yandex AppMetrica или Firebase Analytics для понимания популярности функций.

Локализация – подготовить ресурсы для английского языка (сейчас только русский).

Все решения должны быть совместимы с существующим кодом (использовать текущие модели данных, добавлять новые поля при необходимости).

Код должен быть хорошо структурирован, с комментариями на русском или английском (желательно английском).

Предпочтение отдаётся проверенным библиотекам и подходам.

Учитывать особенности Android-разработки (жизненный цикл, WorkManager, ограничения фоновой работы).
